暂存区和工作区

工作区 （Working Directory）进行工作时所在的文件夹（上文的 gitLearn目录），工作区下有一个隐藏的目录 .git ，这个不算工作区，而是Git的版本库 。

版本库中最重要的就是称为stage（或者叫index）的暂存区，还有Git为我们自动创建的第一个分支master，以及指向master的一个指针叫HEAD。

第一步是用 git add 把文件添加进去，实际上就是把文件修改添加到暂存区；

第二步是用 git commit 提交更改，实际上就是把暂存区的所有内容提交到当前分支。

提交之后若没有做任何更改，那么工作区便是”干净的“：

跟踪修改

Git跟踪并管理的是修改，而非文件。

git 的 快照 是对该版本所有文件状态的一种”描述”，每次提交 commit 时 git 会生成 快照 来记录这次 commit 时整个项目所有文件的信息。

这个快照并不是这次 commit 时项目中所有文件的拷贝，而是一种索引

每次修改，必须先 git add 到暂存区，才能 commit -m 到分支中。

撤销修改

  • 丢弃工作区的修改（还未 git add ）： git restore file

    此时有两种情况：

    一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；

    一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。

  • 将暂存区撤回到工作区（还未 git commit ）： git reset HEAD file 或 git restore --staged file

  • 从master同时恢复工作区和暂存区： git restore --source=HEAD --staged --worktree [file]

  • 撤销 git commit ： 版本回退，使用 git reset --hard

删除文件

在工作区中删除文件时，如果确实是进行删除，使用 git rm 命令，将文件从工作区中删除，并 git add 到暂存区，而后git commit 到版本库。

先手动删除文件并 git add 和 git rm 效果是一样的。

如果是误删除，则直接版本回退，使用 git reset --hard HEAD^ 。

远程仓库

SSH 注册

Git 的本地仓库管理并不能保障数据的安全（硬盘损坏了只能GG），也不利于团队协作。

远程仓库 应运而生。

github 是全球最大的提供Git仓库托管服务的网站。本地仓库可以很方便地推送到 github 上进行托管，无需担心本地数据丢失。

由于 本地Git仓库和GitHub仓库之间的传输是通过 SSH 加密的 ，所以，需要一点设置：

  1. 创建SSH Key。在用户主目录下，看看有没有.ssh目录，如果有，再看看这个目录下有没有id_rsa 和id_rsa.pub 这两个文件，如果已经有了，可直接跳到下一步。如果没有，打开Shell（Windows下打开Git Bash），创建SSH Key：

    • Keygen 为 Key Generator 的简写形式，就是一般所说的注册机，是为软件注册生成所需的注册码的一种程序。

    • 为什么 GitHub 需要 SSH Key 呢？因为 GitHub 需要识别出你推送的提交确实是你推送的，而不是别人冒充的，而 Git 支持 SSH 协议，所以，GitHub 只要知道了你的公钥，就可以确认只有你自己才能推送。

    而后可以在用户主目录里找到.ssh 目录，里面有id_rsa 和id_rsa.pub 两个文件，这两个就是 SSH Key 的秘钥对，id_rsa 是私钥，不能泄露出去，id_rsa.pub 是公钥，可以放心地告诉任何人。

分支管理

了解分支

在平常开发中，一般都会对应三种环境，本地环境、测试环境、线上环境。开发的基本流程都是先在本地环境开发好,再把代码发布到测试环境测试，最后再发布到线上环境。

对于代码，我们使用 Git 分支进行管理，一般而言会对应两个分支, master 和 dev 。

master 分支主要用于线上发布使用，dev 分支用于平常的开发和测试。除此之外，我们也可以创建多个分支，比如给某次的迭代开发创建一个分支、针对一次代码优化创建一个分支，或者针对一次 Bug 修复，我们可以新建个分支。

情形：

  1. 在 dev 分支上修改了 demo.text ，并提交了修改。
  2. 切换到 master 分支。
  3. 在 master 分支上再次修改 demo.text ，并提交修改。
  4. 在 master 分支上合并 dev 分支。

此时报错：

也可使用 git status 查看冲突：

冲突原因在于：同时在两个分支上更改同一文件，Git 不知道采用谁的修改

这种情况下，Git无法执行“快速合并”，只能试图把各自的修改合并起来。若修改的不是同一文件，则可以直接合并 ，而若修改的是同一文件，就可能发生如上的冲突。

解决办法： 直接在当前分支（master）上决定最终的修改，提交后进行合并即可。

注意，修改并合并后，仅仅是把 master 的修改提交了，dev 分支上的东西仍然未变。

合并分支时，如果可能，Git 会优先使用 Fast-forward 模式来进行合并。如果要强制禁用 Fast forward 模式，Git 就会在 merge 时生成一个新的 commit 。

禁用 Fast-forward ：--no-ff

分支管理策略

Git的特色之一就是可以灵活的建立分支，因为有分支的存在，才构成了多工作流的特色。事实的确如此，因为项目开发中，多人协作，分支很多，虽然各自在分支上互不干扰，但是我们总归需要把分支合并到一起，而且真实项目中涉及到很多问题，例如版本迭代，版本发布，Bug 修复等，为了更好的管理代码，需要制定一个工作流程，这就是通常意义上的 Workflow ，也就是我们常说的分支管理策略。

目前使用度最高的工作流前三名分别是以下三种（排名不分先后）：

  • Git Flow
  • GitHub Flow
  • GitLab Flow

其中 Git Flow 出现的最早，GitHub Flow 在 Git Flow 的基础上，做了一些优化，适用于持续版本的发布，而 GitLab Flow 出现的时间比较晚，所以综合前面两种工作流的优点，制定而成的一个工作流。

在采用 Git Flow 工作流的项目中，代码的中央仓库会一直存在以下两个长期分支：

  • Master
  • Develop（dev）

其中 origin／master 分支上的最新代码永远是版本发布状态。origin／develop 分支则是最新的开发进度。

当 develop 上的代码达到一个稳定的状态，可以发布版本的时候，develop 上这些修改会以某种特别方式被合并到 master 分支上，然后标记上对应的版本标签。

团队的每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了，测试完成后再将 dev 合并到 master 上即可。

为什么不能直接 merge ？

一般而言，个人分支 bob 会比 dev 分支新很多，比如 bob 分支修改了 a，b，c 三个文件。而 dev 之前已经修复了 Bug（可能修改了文件d），那么 bob 和 dev 分支都有修改，就无法直接合并，还要进行其他操作，很麻烦。

注意，cherry-pick 也可能引起冲突！

情形：若你在 dev 改 Bug 时，先后提交了两次： 4ed34 和 233af ，那么你就无法直接 cherry-pick 233af ，因为这两次提交的代码很可能是连续的，你只 cherry-pick 后一次修改的代码，大概率会跑不起来。所以需要依次 cherry-pick 。

多人协作

  • 推送分支：

    git push

    如果只写 git push ，则默认推送本分支到之前绑定的远程仓库的对应分支上去。

    如果没有绑定，则：

    1. git push origin dev ，即为本次推送指定远程仓库为 `origin` ，分支为 `dev` ，但没有持续绑定。
    2. git push --set-upstream origin dev ，指定远程仓库为 `origin` ，分支为 `dev` ，且持续绑定（前提是远程仓库和分支存在）。
    3. git push -u origin dev ，指定远程仓库为 `origin` ，分支为 `dev` ，且持续绑定，若远程分支不存在，则创建并推送。

  • 抓取分支：

    • git pull ，抓取远程分支，并与本地分支合并。
    • git fetch ，此命令会抓取远程分支，但不会与本地分支合并！

      抓取后不合并，哪抓取的远程分支放在哪里了？答案是在 远程仓库副本 下。

      上面的 FETCH_HEAD 是一个版本链接，记录在本地的一个文件中， 指向着目前已经从远程仓库取下来的分支的末端版本 。

      可以使用 git log FETCH_HEAD 查看 fetch 日志。

      也可以使用 git diff FETCH_HEAD 或 git diff HEAD FETCH_HEAD 来比较抓取的分支和本地对应的分支的区别，前者是当前分支相对于 FETCH_HEAD 的区别，后者相反。 （个人感觉后者更直观）

    有时 push 代码到 git 时，出现提示：

    问题分析：别人上传到远程仓库后，你没有及时的同步（拉取）到本地，但是你同时又添加了一些内容（提交），以致于你在提交时，它会检测到你之前从远程仓库拉取的时候的仓库状态和现在的不一样。于是，它为了安全起见拒绝了你的提交。

    问题解决：
    （1）先把git的东西fetch到你本地然后merge后再push

    先抓取远程仓库的更新到本地，然后与你的本地仓库合并（可能有冲突，需要先解决冲突才能合并），这样就可以使远程仓库和你本地仓库一致了，然后就可以提交修改了。

    （2）这2句命令等价于

    但是使用git fetch + git merge 更加安全。

    git pull = git fetch + git merge

    首先，基于本地的 FETCH_HEAD 记录，比对本地的 FETCH_HEAD 记录与远程仓库的版本号，然后 git fetch 获得当前指向的远程分支的后续版本的数据，然后再利用git merge将其与本地的当前分支合并。
